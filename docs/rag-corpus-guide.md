# Fraktl — RAG Corpus Guide

> Related: [Architecture](./architecture.md)

The RAG corpus is Fraktl's primary competitive advantage. It grounds biosemiotic narratives in real botanical data and Fraktl's proprietary species signatures — preventing generic AI hallucination.

---

## How It Works

```
seed_rag.py runs once (or on cold start)
  ↓
WikipediaLoader fetches botanical articles per species
  ↓
RecursiveCharacterTextSplitter chunks text (500 tokens, 50 overlap)
  ↓
fraktl_base.json entries added as priority documents
  ↓
OpenAI text-embedding-3-small embeds all chunks
  ↓
ChromaDB stores vectors in /app/chroma_db (persistent Railway volume)

At scan time:
  species name + structural data → ChromaDB similarity search (k=4)
  → top 4 chunks injected into narrate.py prompt
  → GPT-4o generates grounded narrative
```

## Corpus Sources

| Source | Content | How indexed |
|---|---|---|
| Wikipedia | Species morphology, ecology, history | `WikipediaLoader` (1 article/species) |
| fraktl_base.json | Proprietary fractal signatures, Fibonacci notes, energetic narratives | Direct `Document` objects |
| GBIF (future) | Regional observations, growth patterns | `JSONLoader` (custom HTTP fetch) |

## fraktl_base.json Schema

```json
[
  {
    "species": "Quercus robur",
    "signature": "...(2-4 sentences describing the tree's fractal/energetic character)...",
    "fibonacci_note": "...(specific Fibonacci/golden ratio observation for this species)..."
  }
]
```

**File location:** `backend/app/rag/corpus/fraktl_base.json`

### Adding a new species

1. Open `backend/app/rag/corpus/fraktl_base.json`
2. Add a new entry following the schema above
3. Re-seed the corpus:
   ```bash
   cd backend
   python scripts/seed_rag.py
   ```
4. The new species is now retrievable at scan time

### Writing good signatures

- Ground in observable, documented biology (not pure metaphor)
- Reference specific measurements or phenomena (e.g., "ángulo de 137.5°", "espirales 8/13")
- Keep to 2–4 sentences — the full narrative is generated by GPT-4o from this + Wikipedia context
- The `fibonacci_note` should be specific and verifiable

## Seeding the Corpus

### Manual seed

```bash
cd backend
python scripts/seed_rag.py
```

Output: `Seed complete: N chunks indexed.`

### Automatic seed on startup

The backend checks `collection_has_data()` in the FastAPI lifespan. If ChromaDB is empty (first deploy or reset volume), it calls `seed_rag.seed()` automatically before accepting requests.

This adds ~30–60s to cold start time. Normal startup (data already indexed) is instant.

### Forcing a re-seed

```bash
# Delete the local ChromaDB directory and re-seed
rm -rf backend/chroma_db/
python scripts/seed_rag.py
```

On Railway: delete and recreate the persistent volume, then redeploy.

## Growing the Corpus (Phase 2)

In Phase 2, user resonance feedback (`resonance_score 1–5`) enriches the corpus:

```
User rates scan → stored in PostgreSQL scans.resonance_score
Periodic job → fetch high-resonance scans (score ≥ 4)
→ generate "community resonance document" per species
→ add to ChromaDB collection
→ future narratives informed by what resonated with real users
```

This transforms the corpus from a static seed into a living knowledge base — impossible to replicate by competitors using only Wikipedia.
